Project Mission

CareBot is committed to providing an AI-powered platform that helps caregivers, social workers, and families quickly find reliable local resources before they reach a crisis. We crowdsource resource recommendations, moderate submissions, and maintain a high-quality database of Alabama healthcare and social services.

Sprint 3 Goals

Most Sprint 3 effort went into two themes: (1) documentation for the course final (install/deploy, dev setup, user-facing guides, FAQs, test plan, development guide, accessibility research) and (2) accessibility and UI work on public and moderator-facing templates. Application hardening, deployment templates, and final demo artifacts (video, cybersecurity write-up) sit alongside those themes. Bulk moderation (multi-select) was a stretch goal and was not shipped.

Sprint 3 Retrospective

What Went Well

  • Documentation: the full course doc set (user guide, development guide, deploy/dev setup, FAQs, test plan, sponsor guide, final hub) came together on the GitHub Pages site and links back to the app repo.
  • Accessibility & UI: focused passes on templates and flows so public submit and moderation screens better match WCAG-oriented guidance documented in the accessibility research artifact.
  • Deployment examples and security assessment artifacts support a realistic handoff to the next team or sponsor.

What Didn’t Go Well

  • Bulk moderation (multi-select) was planned as a Sprint 3 stretch item but was not implemented—moderators still work one submission at a time.
  • Automated test expansion is still limited by database/migration setup in CI; baseline E2E/unit scope for B10/B11 was closed with the test plan and shipped tests.
  • End-to-end polish required more iteration than expected late in the semester.

Deferred beyond Sprint 3

The following were not completed in Sprint 3 and are left for a future team, sponsor scope, or follow-on course work:

  • Bulk moderation — approve or reject multiple queue items at once (multi-select).
  • Email and SMS notifications — notify submitters when submission status changes (approved, rejected, merged).
  • Provider verification workflow — structured process to confirm provider legitimacy before or after publish.
  • Admin dashboard with analytics — full reporting/analytics beyond the moderation views and charts shipped in-app (deeper metrics, trends, exports).
  • Chatbot integration for approved submissions — tighter coupling between the directory and conversational search using only vetted/approved content (beyond optional chat configured today).
  • Accessibility QA with real older-adult users — moderated usability sessions with caregivers (research-backed guidelines were followed; live user testing was out of scope).

See also: Deferred backlog on backlog-tables.html.

Lessons learned

  • Design-first work from Sprint 2 made Sprint 3 documentation faster because the UI was already stable.
  • Writing docs against shipped code catches drift between intention and implementation.
  • Research-backed writing is slower but worth it for the final deliverable.

What We Would Do Differently

  • Start final deliverables earlier (video + assessment) to avoid end-of-sprint rush.
  • Invest earlier in a repeatable test DB strategy for Django tests.
  • Do an early “repo hygiene” pass (clean up structure, remove dead ends, document conventions) before building new features.
  • Avoid relying on a database dump as the primary setup path; prefer repeatable migrations + seed scripts (the dump workflow was consistently painful and non-idiomatic).
  • Define semester-long roles up front (documentation owner, backlog owner, release/deploy owner) instead of leaving duties ambiguous until late.
  • Maintain the backlog and documentation throughout the semester (not just in the final sprint).
  • Plan at the semester level (milestones and integration points), then use sprints to execute—rather than only planning sprint-by-sprint.
  • Increase sponsor communication cadence (more frequent check-ins and earlier alignment on priorities).
