Where work landed (Sprint 1 → 3)

Most of the team’s engineering time went into public submissions, the moderation queue, and merge / deduplication—keeping bad data out and folding duplicates into one vetted listing. Sprint pages hold planning, backlogs, and technical detail; this table only answers where each theme was the main push.

What we built Sprint 1 Sprint 2 Sprint 3
Public submissions — form, validation, caregiver-facing flow Sprint 1 — fixed broken submit path; rate limits, headers, basic hardening so public POST was safe. Sprint 2 — new submit UI, home/base templates, working user/staff login before moderators touch the queue. Sprint 3 — accessibility and template polish on the same flows (labels, contrast, targets).
Moderation queue — approve · merge · reject (single item) Sprint 1 — shipped the queue and all three actions end-to-end; first usable moderator loop. Sprint 2 — queue HTML/CSS and Django wiring for the redesigned experience. Sprint 3 — moderator screens easier to use with a keyboard and screen reader. Bulk multi-select not implemented—still one row at a time.
Merge & deduplication — field-level smart merge + similarity hints Sprint 1 — field-level merge into an existing resource; reject and “approve as new”; core dedup story. Sprint 2 — moderator-facing similarity UX so merges are informed by scores, not guesswork. Sprint 3 — small hardening and doc alignment with how merge behaves in production.
Similarity search — TF‑IDF, embeddings, pgvector Sprint 1 — first similarity pass for “near duplicate” candidates on the moderation path. Sprint 2 — main build: full-catalog TF‑IDF + pgvector, rank_similar_smart, parameterized SQL, moderator dedup tooling. Sprint 3 — operational notes, test/deploy docs; no second large algorithm rewrite.
Docker & deploy — local + cloud demo Sprint 1 — Docker on Mac and Windows; baseline compose for the class. Sprint 2 — GCP VM demo (firewall, secrets, container on the network). Sprint 3 — env templates and run/deploy guidance for a repeatable handoff.
Documentation — runbooks + this site CHANGES / build notes so teammates could boot the stack. README, similarity/schema runbooks in the app repo. Final GitHub Pages hub — finalproject.html, the documentation hub, and linked guides.
