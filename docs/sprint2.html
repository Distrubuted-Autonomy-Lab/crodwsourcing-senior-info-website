Project Mission

CareBot is committed to providing an AI-powered platform that helps caregivers, social workers, and families quickly find reliable local resources before they reach a crisis. We crowdsource resource recommendations, moderate submissions, and maintain a high-quality database of Alabama healthcare and social services.

Problems We Are Solving

  • Resource discovery: Caregivers and families struggle to find local healthcare resources when they need them.
  • Data quality: Community-submitted resources need validation, deduplication, and structured merge.
  • Rural access: Gaps exist in counties with limited or scattered services.
  • Trust & moderation: Submissions must be reviewed and integrated in a controlled, auditable way.

High-level features (CareBot)

  • Django (PostgreSQL, PostGIS, pgvector): AI chatbot (Gemini), user/staff login working, public resource submission, moderation queue with full-catalog similarity (TF‑IDF + vector embeddings), approve / merge / reject, accessible crowdsourcing UI (sponsor: large targets, readable type, clear controls — owl / out-of-scope branding unchanged).
  • Dev & ops: Docker Compose (Mac + Windows), continued security (CSRF, parameterized SQL, secrets in env), GCP cloud deployment working (Compute Engine VM, firewall, Docker) for course demos, runbooks (Docker, schema fixes), documentation on this site and in the CareBot repo.

Sprint 2 Goals

Sprint 2 focuses on production-quality UX for the crowdsourcing portion of the site (sponsor: redesign is OK; do not change the owl / out-of-scope branding). Use clear patterns for buttons vs. checkboxes vs. dropdowns; keep flows accessible and easy for non-technical users (large touch targets for shaky hands, very readable text). The team shipped working user/staff login (with home and base layout). Similarity uses vector embeddings as the AI approach for comparing listings. Sprint 2 also continues security hardening, delivers a working GCP cloud demo (VM + Docker + firewall), and testing / documentation. Work integrates through PR review and scrum.

1. Similarity search: full database + vector merge

Problem: Similarity only considered rows in resource_embeddings. Bulk-imported resource_listings (and anything never embedded) did not appear, so results looked like “only submitted/approved items.”

Solution (recommendation/similarity.py):

PieceWhat it does
rank_similar(..., top_k=None)TF‑IDF cosine similarity over every corpus document (not only top‑k).
_resource_listing_corpus_text()Rich text per listing: title, description, tags, eligibility, cost, languages, hours, resource type, provider name/subname, address, city, state, zip.
pgvector_similar()Nearest neighbors in Postgres using resource_embeddings + join to resource_listings / providers.
rank_similar_smart()Loads all ResourceListing rows, TF‑IDF scores each, fetches pgvector scores for embedded rows, min–max normalizes, blends: if both signals → 0.35 × lexical + 0.65 × vector; else max(lexical, vector). Returns {rid, title, score}.

Call sites: moderation_detail, public_submit, similar_search — all use rank_similar_smart(query, top_k=…).

Submissions: Approved/merged rows are normal listings in the full scan. Pending items match via query text from the form/moderation.

Ops: Teammates can run python manage.py reembed in the web container to backfill embeddings; lexical search works without it.

3. Schema / teammate runbook (resource_listings.title)

Symptom: ProgrammingError: column resource_listings.title does not exist on /manage/public_submit/ (POST).

Cause: Older SQL dumps; Django expects title. Initial migration does not patch existing tables.

Fix (Postgres):

docker compose -f docker-compose.yml -f docker-compose.local.yml exec postgis \
  psql -U carebot -d carebot -c \
  "ALTER TABLE resource_listings ADD COLUMN IF NOT EXISTS title varchar(255) NOT NULL DEFAULT '';"

Or reset volumes / restore a dump that matches current schema (local data loss).
