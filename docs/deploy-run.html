How to run & deploy the application

Run instructions for developers and demo operators. Canonical code lives in the application repository; this page summarizes the expected workflow for someone new to the project.

Repositories

  • App repo: carebot-deployable-crowdsourcing
  • Website repo: website-crowdsourcing

Run locally (recommended: Docker Compose)

From the app repo’s root/ directory:

docker compose -f docker-compose.yml -f docker-compose.local.yml up --build
  • App: http://localhost:8000/
  • Moderation / staff tools: typically under /manage/… (example: /manage/moderate/)

Common pitfall: run Compose from root/ (wrong working directory breaks paths and env files).

Required configuration (environment variables)

Set secrets in environment files (for example .env-local) and never commit real credentials.

  • Django secrets: SECRET_KEY, DEBUG (should be False in production), ALLOWED_HOSTS, CSRF_TRUSTED_ORIGINS
  • Database: Postgres connection variables used by Compose / Django settings
  • AI integrations (optional / feature-dependent):
    • GOOGLE_API_KEY (Gemini chat)
    • OPENAI_API_KEY (if using OpenAI for embeddings)
    • HF_TOKEN (optional; helps with model download rate limits if using Hugging Face)

See also: FAQs — component services (APIs, OAuth).

Staff login / OAuth

Current staff authentication is standard Django session-based login (team-managed staff accounts). OAuth / “Sign in with Google” is not required unless your sponsor requests it.

If OAuth is added later, document provider setup, redirect URIs, and environment variables under FAQs — OAuth.

Deployment notes (demo / cloud)

  • Recommended: deploy via Docker on a VM (e.g., GCP Compute Engine) using the same Compose files.
  • Network: open only required ports; enforce HTTPS via reverse proxy if internet-exposed.
  • Operations: rotate keys; keep backups; monitor API quotas for third-party services.
