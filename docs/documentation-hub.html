Run & deploy

Running in your own environment, Docker, configuration, and external services (including free vs paid notes).

Primary page: How to run & deploy

If someone deploys on their own environment, what should they do?

  • Clone carebot-deployable-crowdsourcing, configure .env-local (or production env) from the example files in root/.
  • Run with Docker Compose from root/ using docker-compose.yml plus the appropriate override (e.g. docker-compose.local.yml or docker-compose.gcloud.yml for cloud-shaped deploys).
  • Apply migrations inside the app container; open only required ports; use HTTPS in front if exposed to the internet.

What software is needed? (Docker)

  • Docker + Docker Compose bundle the app, PostgreSQL/PostGIS, optional RabbitMQ, and nginx so installs are repeatable across machines.
  • Python dependencies are installed in the image from requirements.txt (see app repo root/).

What external resources are used?

  • Database: PostgreSQL with PostGIS (in Compose or managed Postgres that supports PostGIS).
  • Optional messaging: RabbitMQ if async features are enabled.
  • AI / APIs (optional): Google Gemini (GOOGLE_API_KEY) for chat; OpenAI or other providers for embeddings if configured; Hugging Face model downloads may use HF_TOKEN to ease rate limits.

Free vs paid tiers (typical)

  • Google AI / Gemini: usage is account- and tier-dependent; set quotas in Google Cloud and monitor billing.
  • OpenAI: paid API usage by token; store keys in environment only.
  • Hugging Face: community tier for models; rate limits apply — HF_TOKEN optional.
  • Hosting: GCP/AWS VM costs depend on machine size; Docker keeps parity with local.

API keys and OAuth workflow (syllabus)

  • Required / common keys: database credentials, Django SECRET_KEY, and any keys your deployment uses for optional services — listed on Run & deploy and in the app repo .env examples.
  • OAuth: this deployment uses Django username/password + sessions for staff; there is no OAuth server workflow configured for moderation login. If you add OAuth later, document client IDs, redirect URIs, and token storage in the FAQs and runbook.
  • Optional APIs (e.g. chat): FAQs — which keys the app uses (Gemini is chat-only; crowdsourcing core does not require it).

Staff / sponsor overview Sponsor user guide

Moderation, merge, duplicate handling at a high level.

Features and how they link (examples)

  • Public submission → records appear in the moderation queue for staff; staff approve, merge with an existing directory entry, or reject. Nothing appears in the public directory until approved.
  • Similarity search (home / search tool) helps users and staff find near-duplicate or related resources before submitting; results tie back to directory entries and moderation decisions described in Sprint 2 technical notes.
  • Staff accounts: Django session-based login for moderators (not the same as public submitters). OAuth is optional and documented under FAQs if added.
  • Optional chat (if enabled) uses configured API keys; if disabled, the rest of the crowdsourcing flow still works.

How are external resources incorporated?

Embeddings, similarity, and optional chat call out to providers configured via environment variables; the database holds canonical resource records. See FAQs and Run & deploy.

Development (extend/modify)

After a local install: languages, build, dependencies, backlog, style, and tests.

Full guide: Development guide · DevelopmentGuide.md

  • Languages: Python (Django), HTML/CSS/JS in templates and static, SQL (Postgres/PostGIS).
  • Build / run: Docker images built from the app Dockerfile; manage.py for migrations, tests, and admin.
  • Dependencies: requirements.txt in app root/; pinned in the image build.
  • Automated builds: CI depends on team setup; run tests locally with python manage.py test (see test plan for scope).
  • Backlog / issues: Full project backlog (B1–B14) and deferred themes on the backlog tables page; major migrations (e.g. Django major version, API upgrades) should be tracked as issues or in repo docs.
  • Style expectations: follow existing patterns in the repo; prefer small, reviewable PRs; keep secrets out of git (see cybersecurity assessment).
