2. Which APIs and keys does the app require?

Google Gemini (GOOGLE_API_KEY)

  • Used for: The public chat flow calls the Gemini API (via google.generativeai) to interpret questions and produce responses grounded in query logic and database results.
  • Configuration: Set GOOGLE_API_KEY in the environment (e.g. .env-local — never commit). Optionally set GEMINI_MODEL (default in code: models/gemini-2.5-flash).
  • How to obtain a key: In Google AI Studio or Google Cloud, create an API key with access to the Generative Language API. Restrict the key by HTTP referrer or IP where possible.
  • Considerations: Cost and quotas — monitor usage; set budgets and alerts. Privacy — user queries and model context may leave your server; align with sponsor policy and any HIPAA/FERPA constraints before sending sensitive data. Availability — outages or rate limits degrade chat; fail gracefully in the UI. Model changes — changing GEMINI_MODEL can change behavior, latency, and pricing — test after upgrades.

OpenAI (OPENAI_API_KEY)

  • Used for: Optional paths (e.g. embedding backends in settings). The codebase reads OPENAI_API_KEY; not every deployment needs it, depending on EMBEDDING_BACKEND and active code paths.
  • Configuration: Set the key if you enable OpenAI-backed embeddings or related features.
  • Considerations: Same themes as Gemini — billing, data handling, key rotation.

Django SECRET_KEY

  • Used for: Signing sessions and CSRF tokens.
  • Configuration: Long random string via environment. Never commit real values or reuse keys across untrusted environments.

Database credentials

  • Used for: Django database settings (DATABASE_NAME, DATABASE_USER, DATABASE_PASSWORD, host, port).
  • Considerations: Strong passwords in production; restrict network access; prefer secrets managers or platform env injection over plain files on servers.

Optional integrations

  • Settings may define secrets such as TWILIO_WEBHOOK_SECRET for webhooks — set only if those endpoints are implemented and exposed.
  • Hugging Face: If embeddings download models from Hugging Face, an optional HF_TOKEN can raise rate limits; document in deploy runbook.

3. OAuth: does CareBot use it?

Current approach: Staff and moderation flows use Django’s built-in authentication (username/password, sessions, django.contrib.auth), typically with LOGIN_URL at /accounts/login/. This is not OAuth 2.0 / OpenID Connect against Google, Microsoft, or similar in the default course deployment.

Implications:

  • No OAuth client ID/secret or authorization-code flow for end users in the default setup.
  • Session cookies and CSRF protection are the primary browser security mechanisms for logged-in users.

If the sponsor adds “Sign in with Google” (or similar):

  • Add an OAuth/OIDC provider (e.g. django-allauth, or a cloud IdP) and register redirect URIs, client ID, and client secret.
  • Document provider console steps, scopes, token storage, refresh, logout, and account linking (mapping OAuth identity to existing User rows).

4. Why PostGIS and pgvector?

  • PostGIS: Resource and location data use geospatial types and queries (distance, containment).
  • pgvector: Embeddings live in Postgres for similarity search without a separate vector-only database.
  • Tradeoff: Slightly heavier ops than “Postgres only,” but fewer moving parts than an external vector service.

5. RabbitMQ and Celery

  • RabbitMQ may appear in local Compose for development.
  • Full async worker integration may be partial; CELERY_BROKER_URL must match the environment (container hostname rabbitmq vs localhost).
  • Production: Secure broker credentials; do not expose the management UI publicly without auth and network controls.

6. Selection criteria (summary)

  1. Fit: Gemini for conversational UX; Postgres for relational + geo + vectors.
  2. Cost: Managed API pricing vs. self-hosted LLM.
  3. Security & compliance: Data flows, key custody, audit needs.
  4. Team familiarity: Django + Docker.
  5. Flexibility: Pluggable embedding backend and Celery for future jobs.

7. Periodic review checklist (APIs and auth)

  • API keys rotated after personnel changes or suspected leak.
  • Key restrictions (IP, referrer, API surface) reviewed in cloud consoles.
  • DEBUG=False, SECRET_KEY, ALLOWED_HOSTS, and CSRF_TRUSTED_ORIGINS correct per environment.
  • No secrets in Git history or public READMEs (placeholders only in examples).
  • Rate limiting and abuse monitoring for public chat endpoints remain appropriate.
  • Third-party subprocessors (Google, OpenAI, etc.) still acceptable under sponsor agreements.
  • If OAuth is added: token lifetimes, refresh, and logout documented and tested.

8. Where to look in the codebase

  • API keys and model name: root/app/carebot/settings.py (GOOGLE_API_KEY, OPENAI_API_KEY, GEMINI_MODEL).
  • Gemini usage: root/app/chat/views.py (genai.configure, GenerativeModel).
  • Auth URLs: root/app/carebot/urls.py (django.contrib.auth.urls).
  • Environment template: root/.env-local.example (if present).
