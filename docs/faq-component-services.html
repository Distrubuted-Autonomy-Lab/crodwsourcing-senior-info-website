FAQs: Component services (APIs, OAuth)

Selection, setup, significant considerations, and gotchas for Crowdsourcing Senior Info (CareBot). The same repo ships crowdsourcing (submit → moderation → merge, under recommendation/) and an optional public chat (under chat/). Gemini is only for chat—not required to run the crowdsourcing pipeline.

0. End-user FAQs (quick answers)

These questions are written for public users submitting resources and staff/moderators reviewing submissions (not for developers).

Why don’t I see my submission right away?
Submissions go into a moderation queue first. A staff member reviews it, then approves it as a new listing or merges it into an existing listing.
Who can approve or reject submissions?
Only staff/moderator accounts can access moderation pages. Public users can submit recommendations but cannot directly publish them.
Why does the “similar results” list look wrong sometimes?
Similarity depends on the listing text and (when available) stored embeddings. Some data may not have embeddings yet, and names/addresses may be incomplete—so results can vary. Moderators should still manually confirm duplicates before merging.
Why is chat not responding (or disabled) sometimes?
Public chat is separate from submitting or moderating resources. It only works when operators enable it and set a Gemini API key (GOOGLE_API_KEY). If the key is missing, expired, out of quota, or rate limited, chat may be unavailable—submit and moderation still work without it.

2. Which APIs and keys does the app require?

Crowdsourcing core (public submit, moderation, merge/reject, TF‑IDF / pgvector similarity) runs on Django + Postgres and does not call Google Gemini. GOOGLE_API_KEY is optional and only used by the public chat code in app/chat/.

Required for a typical local or container run

  • Django SECRET_KEY — signing sessions and CSRF tokens. Long random string via environment; never commit production values.
  • Database credentials — Django DB settings (DATABASE_*, host, port). PostGIS / pgvector match the crowdsourcing + similarity features.

Optional — Google Gemini (GOOGLE_API_KEY) for public chat only

  • Scope: Code under root/app/chat/ calls the Gemini API (google.generativeai) for conversational replies. Not used in recommendation/ for submissions or moderation.
  • Used for (when chat is used): Interpreting user questions and producing responses grounded in query logic and database results.
  • Configuration: Set GOOGLE_API_KEY in the environment (e.g. .env-local — never commit). Optionally set GEMINI_MODEL (default in code: models/gemini-2.5-flash).
  • How to obtain a key: In Google AI Studio or Google Cloud, create an API key with access to the Generative Language API. Restrict the key by HTTP referrer or IP where possible.
  • Considerations: Cost and quotas — monitor usage; set budgets and alerts. Privacy — user queries and model context may leave your server; align with sponsor policy and any HIPAA/FERPA constraints before sending sensitive data. Availability — outages or rate limits affect chat only; fail gracefully in the UI. Model changes — changing GEMINI_MODEL can change behavior, latency, and pricing — test after upgrades.

Embeddings / similarity (separate from Gemini)

  • Similarity and merge assistance use Postgres + pgvector and local embedding generation in the app (e.g. recommendation/embeddings.py with SentenceTransformers). That path does not use GOOGLE_API_KEY. Optional cloud keys (e.g. Hugging Face HF_TOKEN for model download rate limits) may apply depending on your deployment—see env templates in the app repo.

Optional integrations

  • SMS submission webhook (Twilio-style): if you use /webhooks/sms/, set TWILIO_WEBHOOK_SECRET and keep the endpoint private (don’t expose without a secret).
  • Email submission webhook: if you use /webhooks/email/, set INBOUND_EMAIL_SECRET and keep the endpoint private.

3. OAuth: is it used in CareBot Deployable Crowdsourcing?

No. In the current carebot-deployable-crowdsourcing deployment, staff/moderation access uses Django’s built-in authentication (username/password + sessions). There is no OAuth/OIDC provider configured, so there is no OAuth client ID/secret to set up for this project.

4. Gotchas we ran into (install + operations)

  • Running Docker Compose from the wrong directory: compose is intended to be run from the app repo’s root/ directory. Running from a different working directory can break relative paths and env-file loading.
  • Django not installed locally: if you try to run python manage.py … on the host without a venv, it will fail. The recommended path is to run Django inside the Docker container.
  • Schema drift from older DB dumps: if you restore an older database snapshot, Django models may not match (errors like missing columns). Fix by applying migrations or resetting local volumes and re-initializing the DB.
  • Staff login confusion: moderation requires a Django user. If no staff account exists yet, create one (e.g. createsuperuser) inside the running container.

5. Why PostGIS and pgvector?

  • PostGIS: Resource and location data use geospatial types and queries (distance, containment).
  • pgvector: Embeddings live in Postgres for similarity search without a separate vector-only database.
  • Tradeoff: Slightly heavier ops than “Postgres only,” but fewer moving parts than an external vector service.

6. RabbitMQ and Celery (not required for core flows)

  • RabbitMQ may appear in local Compose for development.
  • Full async worker integration may be partial; CELERY_BROKER_URL must match the environment (container hostname rabbitmq vs localhost).
  • Production: Secure broker credentials; do not expose the management UI publicly without auth and network controls.

7. Selection criteria (summary)

  1. Fit: Postgres for relational + geo + vectors (crowdsourcing + similarity); Gemini only if you need optional conversational chat.
  2. Cost: Managed API pricing vs. self-hosted LLM.
  3. Security & compliance: Data flows, key custody, audit needs.
  4. Team familiarity: Django + Docker.
  5. Flexibility: Pluggable embedding backend and Celery for future jobs.

8. Possible issues with external resources (and how to handle them)

  • Gemini quota / billing (chat only): if chat stops working, check quota and billing in Google AI Studio / Cloud. Crowdsourcing flows are unaffected.
  • Rate limits / spikes: spikes in traffic can trigger throttling; ensure rate limiting stays enabled on public endpoints.
  • Key leakage risk: never store API keys in Git; rotate keys if exposed. Treat logs/error pages as potential leak vectors.
  • Downtime: third-party outages can break chat only if you rely on Gemini; the chat UI should fail gracefully (safe message, no stack traces).
  • Data handling: do not send sensitive or regulated data to third-party LLM APIs (chat) unless sponsor policy explicitly permits it.

9. Periodic review checklist (APIs and auth)

  • API keys rotated after personnel changes or suspected leak.
  • Key restrictions (IP, referrer, API surface) reviewed in cloud consoles.
  • DEBUG=False, SECRET_KEY, ALLOWED_HOSTS, and CSRF_TRUSTED_ORIGINS correct per environment.
  • No secrets in Git history or public READMEs (placeholders only in examples).
  • Rate limiting and abuse monitoring for public endpoints (including chat, if enabled) remain appropriate.
  • If chat is enabled: third-party subprocessors (Google Gemini) should remain acceptable under sponsor agreements.

10. Where to look in the codebase

  • API keys and model name: root/app/carebot/settings.py (GOOGLE_API_KEY, GEMINI_MODEL, SECRET_KEY).
  • Gemini usage (chat only): root/app/chat/views.py (genai.configure, GenerativeModel) — not used in recommendation/.
  • Auth URLs: root/app/carebot/urls.py (django.contrib.auth.urls).
  • Webhook secrets (if using SMS/email submissions): root/app/carebot/settings.py (TWILIO_WEBHOOK_SECRET, INBOUND_EMAIL_SECRET).
