Final cybersecurity assessment

CS 495 final deliverable — written assessment and periodic review checklist (April 2026).

1) Scope

This assessment covers the CareBot web application (Django), the PostgreSQL/PostGIS database (including pgvector usage), containerized deployment (Docker Compose), and optional third-party API integrations (Gemini / embeddings providers). It assumes a public-facing submission endpoint and a staff-only moderation interface.

2) Assets, trust boundaries, and threats

  • Assets: staff credentials, moderation actions (approve/merge/reject), database contents (resource listings and submitted recommendations), API keys (Gemini/OpenAI/Hugging Face), and server logs.
  • Trust boundaries: public browser → web app; web app → database; web app → external AI APIs; admin/staff browser → moderation endpoints.
  • Threats: credential theft, unauthorized access to staff-only routes, abuse of public endpoints (spam/rate), injection attacks, secrets leakage (in Git, logs, error pages), and data-handling risks when sending text to third-party AI APIs.

3) Controls currently in place

  • Authentication & authorization: staff access via Django authentication (session-based). Moderation actions are not intended for anonymous users.
  • CSRF protection: Django’s CSRF mechanisms are expected for form posts; production configurations should include correct CSRF_TRUSTED_ORIGINS.
  • Secrets management: sensitive configuration (API keys, SECRET_KEY, DB credentials) should be supplied via environment variables; never committed to Git.
  • Input handling: standard Django escaping in templates reduces XSS risk; server-side validation is still required on submission fields.
  • Database safety: vector similarity queries should use parameterized SQL to reduce injection risk; least-privilege DB users are recommended.
  • Abuse resistance: rate limiting and safe error handling were implemented earlier in the project and should remain enabled on public endpoints.

4) Third‑party / AI integration risks

  • Data sent to AI providers: chat and embedding requests may transmit user-entered text to Gemini/OpenAI (or other providers). This must be disclosed to the sponsor and limited to non-sensitive content.
  • Billing and quota: API keys are billable; quotas/rate limits can cause outages if not monitored.
  • Key leakage: API keys accidentally committed to Git or logged by the app is a high-severity risk.

5) Residual risks / gaps

  • Production HTTPS hardening: if deployed publicly, the system must enforce HTTPS end-to-end (reverse proxy, HSTS where appropriate).
  • Centralized logging & monitoring: ensure moderation actions are traceable without logging PII; add alerting for auth anomalies and API quota spikes.
  • Dependency vulnerability management: Docker base images and Python dependencies should be periodically reviewed/updated for CVEs.
  • Spam / misuse: public submission forms can be abused; consider CAPTCHA or additional controls if exposed broadly.

6) Recommendations (actionable)

  1. Enforce HTTPS in production and verify DEBUG=False, correct ALLOWED_HOSTS, and CSRF_TRUSTED_ORIGINS.
  2. Rotate keys (Gemini/OpenAI/SECRET_KEY) on staff turnover or suspected exposure; keep keys out of logs and commits.
  3. Harden staff auth: strong passwords, least-privilege accounts, remove unused staff accounts, and consider MFA if your deployment platform supports it.
  4. Document AI data handling for the sponsor: what data is sent, why, and how it is limited.
  5. Review dependencies quarterly and update base images and Python packages as needed.
