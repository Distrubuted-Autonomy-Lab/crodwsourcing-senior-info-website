2 How to install / deploy software

If someone deploys in their own environment, what should they do? Use Docker Compose from the app root/ with the documented override files, provision PostgreSQL/PostGIS (and pgvector as required), inject secrets via environment (never commit real keys), and open only necessary ports.

  • Software stack: Containers orchestrate the Django app, database, and optional services (e.g. RabbitMQ). Docker is the primary path so builds are repeatable across macOS and Windows.
  • Sources of configuration: docker-compose.yml, docker-compose.local.yml, .env-local (or equivalent), and carebot/settings.py reading environment variables.
  • External resources & tiers:
    • PostgreSQL + PostGIS + pgvector — self-hosted in Compose or managed DB (cost depends on provider).
    • Google Gemini (chat) — API key; usage-based billing / free tiers per Google’s current pricing; monitor quotas.
    • Optional: OpenAI embeddings, Hugging Face model download (HF_TOKEN can help rate limits), GCP (Compute Engine for class demos — VM + disk + egress costs).

3 How to use each completed feature

Features grew across sprints; use the sprint pages for planning context and acceptance themes.

User types Public visitors can submit resource recommendations (crowdsourcing flow). Staff / moderators (Django authenticated users) use the moderation queue, similarity tools, and merge/approve/reject actions.
Typical flow A public submission lands in the moderation queue. Moderators review it, use similarity search to find duplicates, then approve, merge into an existing listing, or reject. Approved items live in the same catalog as imported data (full-catalog similarity in Sprint 2).

Chat (optional): When GOOGLE_API_KEY is set, the public chat can call Gemini; behavior and limits are documented under FAQs — APIs.

4 How to modify / extend software

  • Languages & runtime: Python (Django 5), HTML/CSS/JS templates, SQL (Postgres). No separate “compiler”; the interpreter runs in the container or venv.
  • Build & dependencies: Declared in requirements.txt and Dockerfiles under the app repo. Rebuild images after dependency changes (docker compose up --build).
  • Schema changes: Django migrations (manage.py makemigrations / migrate); coordinate with any legacy SQL dumps (see Sprint 2 title column runbook).
  • Backlog & bugs: GitHub Issues (recommended) or team docs. Major retrofits: Django major upgrades, swapping embedding providers, replacing Gemini, hardening OAuth if added.
  • Style expectations: Follow team conventions (e.g. Black/flake8 if adopted) and document them in CONTRIBUTING.md in the app repo when finalized.

5 FAQs

End users & operators (quick answers)

Why was my submission not visible in search immediately?
Submissions usually wait in the moderation queue until staff approve or merge them into the live catalog.
Why do similar results look weak for some listings?
Similarity blends text and (when present) embeddings. Bulk-imported rows need embeddings backfilled (e.g. reembed); lexical search still covers the catalog — see Sprint 2 technical notes.
Who can log into moderation?
Only Django user accounts your team creates (staff). There is no default “Sign in with Google” unless you add OAuth.
Where do I get an API key for chat?
Operators set GOOGLE_API_KEY in environment — see component services FAQ.

Install gotchas & external services

  • Compose must be run from the correct root/ directory; wrong cwd breaks paths and env files.
  • Schema drift (e.g. missing title on resource_listings) causes 500s until DB matches migrations — see Sprint 2 runbook.
  • Gemini/OpenAI rate limits, billing, and data-handling policies affect production — monitor cloud consoles.
